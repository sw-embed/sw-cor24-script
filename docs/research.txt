You want a small, clean, from-scratch scripting language for an eventual embedded or microprocessor-hosted OS, with these likely requirements:

Interpreter first, not a heavyweight VM

Shell-adjacent use cases: launch binaries, pass arguments, navigate a filesystem, inspect/set environment-like variables

Control flow: conditionals, loops, functions or procedures

Scriptability for system tasks, not primarily numeric/scientific computing

Low implementation complexity

Minimal semantic traps

Reasonable path to future editor/shell integration

Suitable for clean-room design, not “accidentally re-implement JavaScript badly”

That means the right question is not “which language is most popular?” but:

Which semantic model is simplest to specify cleanly?

Which parsing model is easiest to implement?

Which runtime model fits a tiny OS?

Which user experience is best for shell + editor automation?

High-level answer
The simplest clean-room clone to build is usually one of these, in this order:

Tcl-like

Lisp/Scheme-like

Forth-like command language

Lua-like

REXX-like

For your stated goals, my recommendation is:

Best overall for tiny shell scripting: Tcl-like

Best overall for semantic cleanliness and future extensibility: Scheme/Lisp-like

Best overall for familiarity to shell users: REXX-like or a very small Python-like

Best to avoid for a clean-room tiny OS shell: JavaScript-like, Bash/Zsh-like, and probably Python-like indentation syntax

The key design choice
Before discussing languages, it helps to separate the problem into two layers.

Layer 1: command processor / shell
This is the part that understands things like:

cd /tmp

set PATH /bin:/usr/bin

copy a.txt b.txt

run cc hello.c

pipelines/redirection if you add them later

Layer 2: scripting language
This is the part that understands:

variables

if/while/for

function/procedure calls

lists/maps/strings

return codes / exceptions / errors

A major trap is trying to make one syntax do both jobs perfectly. That is why shells get weird.

A cleaner design is often:

shell command language for interactive use

scripting language for nontrivial automation

with easy bridging between them

Examples:

Unix: shell + awk/sed/perl/python

Emacs: commands + Elisp

Tk: commands + Tcl

CMS/TSO: command environment + scripting layer

For your environment, a single language can still work, but the easiest success path is to make it command-oriented first.

What utility is common across most scripting languages?
The common utility is not syntax. It is this:

A command-and-control glue layer
It exists to:

call commands/programs

move strings, lists, and filenames around

branch on success/failure

loop over collections

store small amounts of state

interact with the OS and user

automate repetitive tasks

That is why the best models are those with:

simple value model

simple calling convention

simple error model

simple parser

This immediately favors Tcl, Lisp/Scheme, REXX, and Lua over Bash, JS, or Python.

Candidate languages to consider
1. Tcl-like
This is the one I would put at the top for your use case.

Why Tcl is attractive
Tcl’s core model is astonishingly small:

everything is a command

words are separated into arguments

substitution happens in predictable phases

commands return strings/values

control flow is just commands too (if, while, etc.)

This makes it:

easy to embed

easy to interpret

easy to extend with native commands

naturally shell-ish

far cleaner than POSIX shell

Why it fits your OS
You want:

run binaries

navigate filesystem

environment-like variables

conditional logic

arguments

Tcl already thinks in those terms.

Pros
Very small conceptual core

Parser is much simpler than Bash

Great fit for command dispatch

Easy to expose builtins like cd, ls, run, mount

Easy to treat scripts and shell commands uniformly

No operator-precedence jungle required if you keep expressions small

Cons
Everything-as-string can become awkward

Quoting/substitution rules still need care

Some people find Tcl syntax odd

Less familiar today than Python/Lua/JS

Implementation difficulty
Low to moderate

You need:

tokenizer for words/braces/brackets/quotes

substitution engine

command dispatcher

variable store

list/string utilities

optional expression evaluator

This is much simpler than cloning Bash or Python.

Clean-room clone difficulty
Very good candidate

You can borrow the style without reproducing Tcl exactly.

My verdict
If you want a shell-first scripting language, this is probably the easiest clean design to win with.

2. Scheme/Lisp-like
This is the cleanest semantically.

Why Lisp is attractive
The parser is tiny:

read lists

symbols

numbers

strings

Then evaluation rules do the heavy lifting.

It is very attractive if you already think in terms of interpreters, compilers, and macro systems.

Why it fits your OS
You can make:

(cd "/tmp")

(run "cc" "hello.c")

(if (exists? file) ... ...)

(setenv "PATH" "/bin")

This is easy to specify and easy to implement.

Pros
Small parser

Elegant semantics

Easy macro story later

Easy to host REPL

Great for editor automation too

Very extensible

Minimal syntax ambiguity

Cons
Less shell-like for casual users

Startup friction for users who expect command syntax

Command invocation can feel verbose unless you add sugar

Need to choose scope model carefully

Implementation difficulty
Low

Especially if you keep it small:

dynamic typing

cons lists or vectors

symbols

simple environment frames

native functions

no hygienic macros at first

Clean-room clone difficulty
Excellent

Probably the easiest language family to implement cleanly.

My verdict
If you want the most principled and extensible design, this is the strongest choice.

3. Lua-like
Lua is one of the best practical embeddable scripting languages ever made.

Why Lua is attractive
Small language

Clean semantics

Good embedding model

Pragmatic standard library shape

Tables are powerful

Why it may or may not fit
Lua is better as an embedded scripting language than as a shell language.

You can absolutely build shell commands into it, but its syntax wants to be a language first, shell second.

Pros
Compact and elegant

Very practical

Small runtime by modern standards

Tables reduce the number of built-in data structures needed

Clear control structures

Cons
Parser is more involved than Lisp or Tcl

More language machinery than you may want

Expression syntax/operator precedence increases implementation burden

Less naturally command-oriented

Implementation difficulty
Moderate

Still much simpler than JS/Python, but more work than Tcl or Lisp.

Clean-room clone difficulty
Reasonable

But more moving parts.

My verdict
Excellent if you want an embedded language for applications. Less ideal as the first shell language unless you really want a language-first system.

4. REXX-like
This is underappreciated for systems scripting.

Why REXX is attractive
REXX was designed for scripting, command execution, string handling, and OS glue.

It is:

string-oriented

much less insane than shell syntax

Pros

Good for automation and batch tasks

Implementation difficulty
Moderate

Probably easier than Python/JS, harder than Tcl/Lisp.

Clean-room clone difficulty
Good, but more care required.

My verdict

10. Bash/Zsh-like
This is the most dangerous inspiration here.

Other scripting languages worth considering
Here are some additional families worth examining briefly.

Icon / SNOBOL descendants
Interesting for strings and pattern matching.

powerful text manipulation

unusual control semantics

probably not the right first shell language

Janus / mu / tiny command DSLs
Many small tools invented minimal scripting syntaxes. Their lesson is:

DOS batch / DCL / CLIST
Useful historical references for command interaction, but mostly examples of where syntax becomes painful if underdesigned.

Pike / Io / Janet
Modernish languages with elegant ideas. Janet in particular is small and Lisp-ish.
But for a clean-room tiny OS, these are more inspiration than targets.

My recommendation: three viable architectures
Option A: Tcl-like core
This is my top recommendation for your exact request.

Shape
command-oriented syntax

strings + lists as primary values

commands return integer/string/list

variables are dynamically typed

builtins for filesystem/process/environment

control constructs are just commands

Example
set src "hello.c"
if {exists $src} {
    run cc $src -o hello.bin
    if {$status == 0} {
        run ./hello.bin
    } else {
        echo "compile failed"
    }
}
Why this wins
shell-like without shell insanity

easy to implement

good interactive REPL story

great for OS command integration

Option B: Lisp/Scheme core with shell-friendly sugar
This is my favorite if you care most about language cleanliness.

Shape
s-expression interpreter

commands are functions

add convenient command aliases

possibly support one-line shell-ish forms in the REPL

Example
(let ((src "hello.c"))
  (when (exists? src)
    (let ((rc (run "cc" src "-o" "hello.bin")))
      (if (= rc 0)
          (run "./hello.bin")
          (echo "compile failed")))))
Why this wins
easiest semantic correctness

easiest macro/extensibility path

superb for editor automation later

pairs well with your existing language tooling mindset

Risk
Users may want a more shell-like experience interactively.

Option C: REXX-like command language
This is strong if you want human-readable system scripting.

Example
src = "hello.c"
if exists(src) then do
    rc = run("cc", src, "-o", "hello.bin")
    if rc = 0 then
        call run "./hello.bin"
    else
        say "compile failed"
end
Why this wins
approachable

procedural

system-oriented

good string handling model

Risk
Slightly more work than Tcl/Lisp to nail cleanly.

exit exits the script/interpreter with a process code

That separation is worth preserving from the beginning.

13. My recommended v0 rules
Top-level script completion
end of file without error → exit code 0

exit N → exit code N

parse/runtime error → interpreter exits with fixed nonzero code like 100 or 101

Running another script as child process
Use:

run sws file.sws
Result:

if sws launches, $rc.run = 0

$rc.prog = child_sws_exit_code

Running another script in-process
Use:

source file.sws
Result:

returns 0 on success

errors propagate as script errors

return
reserved for future

not implemented in v0

not valid at top level

14. Final recommendation
Yes:

a script should run another script as a child with run sws script.sws

sws should return:

explicit exit N if present

otherwise 0 on normal completion

fixed nonzero interpreter error code on parse/runtime failure

separate that from source file.sws, which runs in the same interpreter and returns success/error in-process

That gives you a clean model now and a solid foundation for future procs, returns, and try/catch.
